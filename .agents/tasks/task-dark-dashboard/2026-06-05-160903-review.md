# Dark-themed promo dashboard with sidebar navigation and editable tables

A new route at `/promo-dashboard` adds a full-viewport dark overlay that renders a KALOVA-branded header with brand/month/year selectors, a left sidebar for switching between seven tab views, and an inline-editable table for each tab. The page is entirely client-side with no persistence — data lives in React state and vanishes on refresh. The overlay approach (fixed inset-0 z-50) hides the existing light Navbar without modifying the root layout.

**Watch for:** Data loss on any navigation or refresh (confirmed). The fixed-overlay layout traps the user with no escape route and no integration with the existing app navigation (confirmed). The `globals.css` body rule (`bg-slate-50 text-slate-800`) bleeds through on flash before the component mounts (likely).

## High-level view

The page is a self-contained client component with no persistence layer. State is a `Record<string, string[][]>` initialized with three empty rows per tab. Every cell keystroke triggers a full shallow-copy of the active tab's data — at 18 columns ("Tab Promo"), this will become noticeable as users add rows.

The overlay strategy — a `fixed inset-0 z-50` container — is a workaround for the root layout always rendering the Navbar. It creates a dead-end: there is no link back to the main app and no way to dismiss the overlay. The Navbar and RoleProvider still mount behind the overlay, consuming resources for no purpose.

The header dropdowns (brand, month, year) are purely decorative. Their values are uncontrolled and unused — no state captures the selected month or year, so filtering or scoping table data by period is impossible without further work.

<summary>Issues (5)</summary>

1. **No data persistence** — All table edits are lost on navigation or page refresh. Add localStorage sync or API persistence if data should survive across sessions.
2. **Overlay has no exit** — The fixed z-50 overlay traps users with no link back to the rest of the app. Add a close button or route link.
3. **Uncontrolled header dropdowns** — Brand, month, and year selectors have no associated state or effect. Either wire them to filter/scope the table data or remove them to avoid misleading the user.
4. **Light-body flash** — `globals.css` applies `bg-slate-50 text-slate-800` to `<body>`, which is visible during hydration before the overlay mounts. Override at the layout or body level for this route to prevent flash of light background.
5. **Missing row key stability** — Table rows are keyed by array index (`key={rowIdx}`). Inserting or deleting rows (if delete is added later) will cause React to misassociate input state with the wrong row. Use a stable ID per row.

<details>
<summary>Details</summary>

## Overlay-as-layout and navigation dead-end

The page stacks a `fixed inset-0 z-50` container over the root layout (which renders `<Navbar />` and a light-themed `<main>`). The Navbar, RoleProvider context, and all their links mount behind the overlay — invisible but present and consuming resources.

There is no path out of the overlay. No close button, no navigation link back to `/`, no keyboard shortcut. A user arriving at `/promo-dashboard` is stuck unless they manually edit the URL. The FEAT-002 spec called for the page to "NOT show the default light Navbar" but didn't address return navigation.

A route-group layout (`app/(dark)/promo-dashboard/layout.tsx`) that omits the Navbar entirely would eliminate the hidden mount overhead and make navigation explicit.

## Ephemeral state and no persistence

`initTableData` creates three empty rows per tab on mount. Every keystroke updates state via `handleCellChange`, which shallow-copies the entire active tab's row array. There is no debouncing, no `localStorage` sync, no API call.

Any accidental navigation (back button, browser refresh) destroys all entered data without warning. There is no dirty-state detection or `beforeunload` handler.

The state shape `Record<string, string[][]>` is flat — no row IDs, no metadata. If row deletion is added later, the index-based keys on `<tr key={rowIdx}>` will cause React reconciliation bugs where inputs "stick" to the wrong row.

## Header dropdowns are cosmetic

The three `<select>` elements in the header (brand, month, year) have hardcoded `<option>` lists but no `value` or `onChange` handlers. A user changing "Januari" to "Maret" sees the dropdown change visually but the table content is completely independent. This misleads users into thinking they're scoping their data entry to a specific period.

## Styling conflict with global body rule

`globals.css` declares:

```css
body {
  @apply bg-slate-50 text-slate-800;
}
```

On initial load, the body flashes `bg-slate-50` (light gray) before the React component mounts and paints the `bg-[#0f0f23]` overlay. Since Next.js renders the root layout server-side (including the body class), the slate-50 background is in the initial HTML payload. On slower clients the flash is visible and contradicts the dark theme.

## Table rendering for wide tabs

The "Tab Promo" tab defines 18 columns. The inputs use `w-full` within cells that have no explicit `min-width`, so narrow columns (like "HPP" or "NPM") compress to near-zero width at typical viewport sizes. Adding `min-w-[100px]` or similar to `<td>` elements would ensure usability.
